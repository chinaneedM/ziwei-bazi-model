# 训练例题导入标准

## 1. 用户提交方式

每个案例使用一个独立压缩包。推荐命名：

- `例题N紫微文字盘.txt`
- `例题N八字排盘.png`，也可提供文字或 JSON
- `例题N题目.txt`
- `例题N答案.txt`

文件名称不必完全固定；只要角色标注清楚，导入器按文件名与内容共同分类。RAR、ZIP、7Z 均可接受。

## 2. 自动导入职责

收到训练例题后，系统负责：

1. 机械解包并计算原压缩包和成员文件 SHA256。
2. 识别紫微盘、八字盘、题目和答案。
3. 保留题目及选项原文，不改写字面时态、期间、持续或频率量词。
4. 将八字图片转成结构化字段，并与原图逐字段复核。
5. 将无答案案例写入运行仓库。
6. 将答案对象写入独立私有答案库。
7. 建立不可变案例 ID、组清单、版本、哈希和回执。
8. 完成答案泄漏扫描后，才标记为可开始基线预测。

## 3. 八字图片处理

训练所需权威对象是经原图核验的结构化转录，而不是图片排版本身。至少保存：

- 阴历、阳历、性别及乾坤造
- 年、月、日、时四柱
- 藏干及十神
- 起运和交运
- 大运序列
- 空亡
- 旺相休囚死
- 四柱神煞
- 天干、地支合冲刑害破及其他可见关系
- 原图 SHA256、尺寸和字节数

原始图片是否写入 Git 为可选项；只要结构化内容逐字段核验通过，即可用于训练。出现任何不确定字、缺失字段或图片与转录不一致时，案例进入 `HOLD_AND_CORRECT_BEFORE_TRAINING`，不得静默猜测。

## 4. 答案隔离

包含答案的原始压缩包只允许在临时导入工作区解包。运行仓库不得保存答案向量、正确选项或答案正文。

- 运行仓库：`training-data/{group_id}/cases/`
- 私有答案库：`answers/{group_id}/`

导入会话已经看过答案，因此不得在同一会话执行盲态基线预测。正式预测必须在新的答案盲态会话中，只读取运行仓库对象。

## 5. 题目与答案校验

每个案例必须满足：

- 机械解析题数与原文题数一致。
- 答案向量长度等于题数。
- 每个答案符号属于对应题目的合法选项集合。
- 每道题的题干、选项文字和顺序原样保存。
- 无法解析或存在歧义时进入 HOLD，不根据语义猜答案映射。

## 6. 分组与循环训练

默认每5个独立案例组成一个 DEV 组。未满5案时放入待组状态；满5案并完成全部校验后，状态变为 `READY_FOR_BASELINE_PREDICTION`。

组内流程：

1. 五案分别生成盲态预测。
2. 所有基线预测及冻结回执齐全。
3. 组状态才允许揭盲。
4. 答案库执行字面重放和评分。
5. 缺陷修补只进入候选版本及回归流程，不覆盖原预测。

## 7. 重复与修订

- 原压缩包 SHA256 完全相同：复用已有案例引用，不重复创建。
- 名称相同但内容哈希不同：建立新修订或新案例 ID。
- 已提交案例不得覆盖。
- 每个组使用 `HEAD.json` 指向不可变修订。

## 8. 用户日常操作

用户只需继续按“一个案例一个压缩包，文件角色标注清楚”的方式提交。分类、转录、核验、分仓、命名、版本、组建、回执及后续调用由系统负责。
