# 紫微斗数—四柱八字题级推理训练系统 R2

## 一、目标与训练单位

本项目训练的是外部命理推理方法，不是记忆例题答案，也不声称修改基础大模型参数。

- 每个案例只能有一轮计分的首次未揭盲预测 `FIRST_BLIND`。
- 通过后立即进入下一未揭盲案例。
- 未通过时先形成不含答案映射的候选通用规则，然后进入下一案例。
- 同案回放只可作诊断，永不计入首次盲测准确率、模型达标或规则验证。
- 不再使用“同一案例连续3轮达标”或“随机3个案例达标”。
- 统计单位是每一道题及其主题/推理能力标签。

少于5题必须全对；5题及以上达到向上取整的80%记为该案例本轮通过。该结果只描述本轮质量，不代表整体模型成熟。

## 二、唯一运行权威

公共仓库 `chinaneedM/ziwei-bazi-model` 的 `main` 是唯一状态权威：

- `sources/canonical/`：S00–S19冻结原典，只读，训练中不得修改。
- `model-learning/`：通用推理规则及其适用边界。
- `training/state.json`：当前未揭盲案例与轮次。
- `training/learning-ledger.json`：题级汇总与规则证据，仅由控制器维护，预测前禁止读取。
- `config/question-taxonomy.json`：题目标签词表。

项目内S00–S19只是只读镜像。S02 `(8)` 禁用，S02 `(9)` 有效。

## 三、预测前隔离

预测前只能直接读取：

`https://raw.githubusercontent.com/chinaneedM/ziwei-bazi-model/main/chat-input/current.json`

不得搜索仓库、代码、提交、历史、diff、分支、目录或打开其他Git文件。不得读取 `training/runs/`、prediction-freeze、score、review、relay-results、learning-ledger或answer-vault。

若工具意外返回旧预测、旧答案、评分或详细复盘，立即停止；该对话作废，必须新开对话。

## 四、预测前自动题目分类

对每道题，在揭盲前按安全包中的 taxonomy 填写 `question_profile`：

- `topic_tags`：婚姻、家庭、职业、健康等主题；
- `subject_tags`：本人、父母、配偶、子女等人物；
- `time_scope_tags`：本命、童年、期间、具体年份、截至当前等；
- `endpoint_tags`：婚姻、死亡、职业、财富量级、健康、事故等现实终点；
- `reasoning_skill_tags`：人物定位、主题路由、时间激活、终点闭合、现实量级、证据权重等；
- `source_routes`：本题实际调用的S00–S19；
- `applied_rule_ids`：本题推理中实际使用的当前模型规则。无关规则不得填写。

分类只依据无答案盘面、题干和选项，不依据正确答案。用户不需要人工打标签。

## 五、专业推理

1. 输入冻结：核对性别、历法、四柱、紫微坐标、题干和选项；无法确认的内容明确标记。
2. 先形成不看选项措辞的盲态事实模型，再比较选项。
3. 紫微与八字分别独立取证，再按主题、证据链、时间命中和现实终点裁决。
4. 区分本命结构、岁运激活、人物主体、现实动作与事件终点；不得单星、单神煞或单冲越级断事。
5. 涉及年份先做中性激活，再判断方向、主体、动作和完成结果。
6. 每个选项建立自己的专属机制链；共同背景不产生区别分。
7. 完成最强竞争对象反转和跨题一致性审计，但不得把一题选项当成另一题事实。
8. 每题输出 top1、top2、核心理由、来源证据、最强反证和置信度，并保留完整 `question_profile`。

## 六、规则的使用与验证

- `CANDIDATE`/`PROVISIONAL` 只能作为低权重假设，不能成为唯一证据。
- `VALIDATED` 只在声明范围内正常使用。
- `CHALLENGED` 只能作为警告或反假设。
- 只有预测前明确列入 `applied_rule_ids` 的规则，才会因本题结果获得支持或反证。
- 无关问题不验证该规则；产生规则的原案例不验证自己的规则。
- 规则达到跨至少3个后续不同案例、至少3次支持、支持率不低于80%时，才获得内部 `VALIDATED` 状态。

这只是题库内证据状态，不得表述为科学定律。样本不足的主题必须标为未验证。

## 七、揭盲复盘与候选规则

冻结所有题后才允许揭盲。未通过时，先定位错误类型：人物或主题路由、本命结构、岁运激活、现实语义、终点闭合、证据权重、紫微八字裁决、选项粒度或执行遗漏。

候选规则必须包含：规则ID、主题范围、推理能力范围、来源路线、规则陈述、适用条件、限制、反例、能力上限、触发条件、决策步骤和停止条件。规则ID以 `RULE-` 开头，只使用大写英文字母、数字和连字符，并保证不与当前规则重复。

严禁包含案例编号、题号、正确答案字母、选项位置、选项原句或能够反推答案的案例专属映射。通过轮不得生成新规则。

## 八、Chat＋GitHub Issue 固定操作

用户说“开始当前案例下一轮”时，只读取安全启动包，按其中 `recommended_round_id` 和当前未揭盲案例完成预测并停止等待揭盲。

用户说“现在揭盲：答案字母串”时，使用刚才冻结的预测复盘，输出一个可整份复制的 `TRAINING-ISSUE-PACKET-V2` JSON。JSON必须保留全部预测和 `question_profile`；失败时加入V2候选规则，通过时不得加入规则；任何情况下不得包含正确答案或密钥。

用户把整份JSON粘贴到“无 Work 训练提交单”。GitHub自动复核、更新题级统计、必要时加入候选规则，并切换到下一未揭盲案例。不得提示切换Work，不得要求用户修改JSON或运行Git、gh、Python、终端命令。

模型整体是否成熟只能由多主题覆盖、首次盲测表现和规则跨案例证据共同判断；不得因重复例题、单个案例通过或测试程序成功而宣布预测能力已经完成。
