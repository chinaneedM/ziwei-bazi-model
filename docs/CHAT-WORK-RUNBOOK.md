# 日常训练操作单：Chat＋GitHub Issue

> 当前107例题库仍处于`DATASET_FROZEN_AWAITING_ANSWER_IMPORT`，本操作单暂不执行。只有安全包显示`prediction_allowed=true`后才开始。

## 一、一次性准备

1. 项目中的S00–S19保留为只读镜像；S02 `(8)` 停用，只使用 `(9)`。
2. 用 `docs/PROJECT-MAIN-PROMPT-R2.txt` 完整替换项目旧指令，不要追加混用。
3. 各案例的加密答案与 `FORTUNE_ANSWER_KEY` 已准备后，日常无需再处理密钥。
4. Chat必须能在同一项目中只读检索完整S00–S19；如果项目来源不可用、缺库或版本不符，停止预测，不凭启动包中的摘要代替正文。

## 二、开始当前案例下一轮

必须新开Chat，原样发送：

> 开始当前案例下一轮。只能直接读取安全启动包 `https://raw.githubusercontent.com/chinaneedM/ziwei-bazi-model/main/chat-input/current.json`，不得搜索GitHub仓库、代码、提交、历史、diff、分支、目录或读取其他Git文件。先确认 `prediction_allowed=true`，并确认项目S00–S19只读镜像均可检索；否则停止。按包内 taxonomy 先为每题建立完整 `question_profile`，再使用当前无答案案例、23张知识卡、当前模型规则和项目来源，从零完成 `recommended_round_id` 的全部预测。每题按`rule_router`最多调用6条范围匹配规则，并用`rule_attribution`区分决定性、辅助性和反证规则。逐题输出top1、top2、核心理由、来源证据、最强反证、置信度、question_profile及rule_attribution。不得读取旧预测、答案、评分、复盘或learning-ledger。完成后冻结并停止，等待我揭盲。

每个新案例只计一次首次盲测，闭环后进入下一新案。3个不同新案连续达标才通过阶段门；失败案至少间隔5个新案后复训。复训必须从零重做推理，但不计首次盲测或晋级证据。

## 三、Chat→Work机器交接

全部预测冻结后，Chat必须立即创建且只创建一个GitHub Issue：

- 标题严格使用安全启动包`chat_work_handoff_contract.issue_title`；
- 正文Schema为`CHAT-WORK-PREDICTION-HANDOFF-V1`；
- 正文直接使用安全启动包的`handoff_payload_template`，只填入完整冻结`predictions`，不手工重抄`binding`；
- 不得包含答案、正确选项、评分、复盘、`expected_result`或任何学习修正。

Work开始闭环时先读取`main/chat-input/current.json`，再读取该唯一交接Issue并逐项核对案例、轮次、模型发布和全部绑定哈希。不得从对话记忆或摘要重建冻结预测。交接缺失、重复、已过期或任一绑定不符时，在评分前停止，不创建`[TRAINING ROUND]` Issue。

用户不再返回Chat创建交接或生成第二份提交单。Chat必须在第一次提示切换Work之前自行创建有效交接；Work通过一次性公钥触发私密评分，并在同一个Work会话内完成PASS提交，或完成FAIL复盘、通用修正与提交。

## 四、揭盲

预测完成后，在同一个Chat发送：

> 现在揭盲：在这里填写答案字母串。严格使用刚才冻结的全部预测评分和复盘，生成可整份粘贴到“无 Work 训练提交单”的 `TRAINING-ISSUE-PACKET-V2` JSON。必须保留每题question_profile；若失败，生成不含案例编号、题号、答案字母、选项位置和选项原句的候选通用规则；若通过，不生成规则。JSON不得包含正确答案或密钥。

然后：

1. 复制Chat输出的整个JSON代码块。
2. 打开仓库“无 Work 训练提交单”。
3. 正文框 `Ctrl+A`，再 `Ctrl+V`。
4. 直接提交Issue。

GitHub会自动复核答案：新案通过则累加跨案连续次数并进入下一新案；新案失败则归零，激活通用模型修正、加入间隔复训队列后进入下一新案。同案复训不改变跨案连续次数。

## 五、系统自动处理的内容

- 每题主题、人物、时间、终点、推理能力和来源标签；
- Top1正确率和Top2覆盖率；
- 各主题与推理能力的样本量；
- 本题实际使用了哪些候选规则；
- 不同新案连续达标次数与间隔复训队列；
- 失败后通用规则的安全检查与激活；
- 当前或下一案例的安全启动包刷新。
- 每25题短维护、每100题中期维护与异常触发维护；
- 规则归并/抑制、每题动态路由、决定性归因、置信度校准和复训成效账本。

用户不需要人工分类题目、不需要判断哪些题能验证哪些规则，也不需要运行Git、gh、Python或终端命令。

## 六、重要边界

- 新案通过：跨案连续次数加1，进入下一新案，不生成答案导向规则。
- 新案失败：跨案连续次数归零，生成并激活通用候选修正，加入间隔复训队列后进入下一新案。
- 3个不同新案连续通过：阶段泛化门达成，但仍按日程继续覆盖案例。
- 不相关的后续题目不会验证候选规则。
- 同案复训不计新的首次盲测准确率、独立规则证据或阶段门。
- 样本不足的领域只显示“未验证”，不能用其他主题成绩替代。
- 当现有案例全部完成时，需要增加新的无答案案例及对应加密答案，系统才能继续积累跨案例证据。
