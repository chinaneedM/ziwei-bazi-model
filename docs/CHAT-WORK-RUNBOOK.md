# Chat / Work 日常训练操作单

## 一、第一次准备答案

这一步只做一次。把每个案例的官方答案明文留在仓库外，由 Work 执行：

```bash
fortune-train keygen
FORTUNE_ANSWER_KEY='保存好的密钥' fortune-train encrypt-answer DEV-EXAMPLE-001 /tmp/DEV-EXAMPLE-001.answers.json
```

对 5 个案例依次生成加密答案并提交公共仓库。明文答案与密钥绝不提交。

## 二、在 Chat 中开始一轮

把下面这段直接发给 Chat：

> 继续“半自动命理预测训练系统”。本轮只训练当前状态中的单个案例。预测前只允许读取 Git 仓库 `sources/canonical/`、当前 `model-learning` 发布和无答案案例；不得读取答案、密钥、旧轮次逐题揭盲或旧选项映射。按控制器当前轮次作答全部选择题，给出每题 top1、top2、理由和证据，先不要揭盲。

Chat 完成预测后，把预测内容交给 Work。Work 负责创建轮次、冻结和评分，用户不需要输入 Git 命令。

## 三、达标时

- 若连续达标次数是 1 或 2：仍训练同一案例，再开新轮次。
- 若连续达标次数是 3：当前案例完成，自动进入下一案例。

## 四、未达标时

在已经揭盲的 Chat 对话中发：

> 对本轮错误做深度复盘。区分输入识别、取象、结构、应期、现实语义、双体系裁决、选项比较和执行遗漏。禁止记忆案例编号、题号、答案字母、选项位置或独特原句；禁止改写用户原典。只产出可推广的模型思路、知识运用方法、执行步骤或待验证假设，并写明适用条件、限制、反例、能力上限与所关联的 S00–S19 模块。

然后切到 Work，发：

> 把刚才复盘形成的通用模型修正写入当前公共 Git 仓库的 `model-learning/`，运行防污染验证，激活为新的模型发布，提交并发布；不得修改 `sources/canonical/`。完成后为同一案例开启下一轮。

Work 会完成文件修改、测试、Git 提交和发布。若修正含案例专属答案映射，控制器会拒绝。

## 五、为什么不能全在 Chat 自动落库

Chat 可以完成命理推理和复盘，但当前产品边界下，GitHub 插件和仓库写入只在 Work/Codex 中可用。纯 Chat 不能可靠地提交 Git 修改。为了省额度，Work 只用于每次失败后的短暂落库，以及必要的系统修复；预测和复盘仍在 Chat 完成。

## 六、用户只需要说什么

日常无需使用 `gh`、Git 或 Python 命令。用户只需说：

> 开始当前案例下一轮。

或：

> 本轮已揭盲，请复盘并把通用修正落库后继续同一案例。

系统不得因为“测试通过”“文件齐全”或“训练次数够多”而宣布案例完成。唯一完成条件是当前案例连续 3 轮达到规定准确率。
