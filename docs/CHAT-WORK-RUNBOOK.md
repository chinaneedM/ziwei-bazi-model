# 日常训练操作单：Chat＋GitHub Issue

## 一、一次性准备

1. 项目中的S00–S19保留为只读镜像；S02 `(8)` 停用，只使用 `(9)`。
2. 用 `docs/PROJECT-MAIN-PROMPT-R1.txt` 完整替换项目旧指令，不要追加混用。
3. 各案例的加密答案与 `FORTUNE_ANSWER_KEY` 已准备后，日常无需再处理密钥。

## 二、开始一个新案例

必须新开Chat，原样发送：

> 开始当前未揭盲案例。只能直接读取安全启动包 `https://raw.githubusercontent.com/chinaneedM/ziwei-bazi-model/main/chat-input/current.json`，不得搜索GitHub仓库、代码、提交、历史、diff、分支、目录或读取其他Git文件。若 `prediction_allowed=true`，按包内 taxonomy 先为每题建立完整 `question_profile`，再使用当前无答案案例、当前模型规则和项目S00–S19只读镜像，从零完成 `recommended_round_id` 的全部预测。逐题输出top1、top2、核心理由、来源证据、最强反证、置信度及question_profile。不得读取旧预测、答案、评分、复盘或learning-ledger。完成后冻结并停止，等待我揭盲。本项目固定使用Chat＋GitHub Issue，不切换Work。

每个案例只做这一次计分的首次盲测。系统不会要求在同一案例上重复三轮。

## 三、揭盲

预测完成后，在同一个Chat发送：

> 现在揭盲：在这里填写答案字母串。严格使用刚才冻结的全部预测评分和复盘，生成可整份粘贴到“无 Work 训练提交单”的 `TRAINING-ISSUE-PACKET-V2` JSON。必须保留每题question_profile；若失败，生成不含案例编号、题号、答案字母、选项位置和选项原句的候选通用规则；若通过，不生成规则。JSON不得包含正确答案或密钥。

然后：

1. 复制Chat输出的整个JSON代码块。
2. 打开仓库“无 Work 训练提交单”。
3. 正文框 `Ctrl+A`，再 `Ctrl+V`。
4. 直接提交Issue。

GitHub会自动复核答案、记录题级统计、更新规则证据并进入下一个未揭盲案例。

## 四、系统自动处理的内容

- 每题主题、人物、时间、终点、推理能力和来源标签；
- Top1正确率和Top2覆盖率；
- 各主题与推理能力的样本量；
- 本题实际使用了哪些候选规则；
- 相关规则在未来不同案例中的支持与反证；
- 失败后候选规则的安全检查；
- 下一未揭盲案例和安全启动包刷新。

用户不需要人工分类题目、不需要判断哪些题能验证哪些规则，也不需要运行Git、gh、Python或终端命令。

## 五、重要边界

- 通过：直接进入下一案例，不生成新规则。
- 失败：生成候选规则后进入下一案例，不回原题刷分。
- 不相关的后续题目不会验证候选规则。
- 同案回放不计正式成绩。
- 样本不足的领域只显示“未验证”，不能用其他主题成绩替代。
- 当现有案例全部完成时，需要增加新的无答案案例及对应加密答案，系统才能继续积累跨案例证据。
